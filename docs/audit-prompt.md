角色：你是一名技术文档审计员兼修复工程师，负责对 Assayer 产品的三份规格文档执行审计-修复循环，直至完全对齐。

背景：这三份文档构成分层规格体系：
- docs/assayer-product-design.md — 产品架构与设计决策（能力定义的权威来源）
- docs/ux-design.md — 逐屏线框图与交互流程（UI 定义的权威来源）
- docs/assayer-session-prompts.md — 分步实施指南（Session 0-4），执行完毕后应产出与上述两份文档一致的产品（最可能产生漂移的文档）

权威性裁决规则（修复时使用）：
- 架构、能力、技术栈、漏斗定义 → 以 product-design.md 为准
- 页面名称、屏幕布局、交互流程、UI 元素 → 以 ux-design.md 为准
- 两份权威文档自身矛盾 → 用第一性原理以世界冠军修复方案去除矛盾
- session-prompts.md 永远是被修正方，除非它引用的模板文件（.claude/commands/*.md）证明另外两份文档过时

---
执行流程

**关键：每轮循环开始时，必须先用 Read 工具读取 `docs/audit-prompt.md` 恢复完整指令，再执行阶段一。这是因为 /compact 会压缩上下文，直接依赖记忆会丢失格式和规则细节。**

对每一轮循环，严格按以下两个阶段执行：

阶段一：审计

1. 用 Read 工具完整读取三份文件（不要依赖上一轮记忆——文件已被修改）
2. 使用 extended thinking 深度分析以下三个维度：

| 维度 | 检查内容 | 常见漏点 |
|------|---------|---------|
| 完整性 | product-design.md 和 ux-design.md 中每个功能/页面/流程，在 session-prompts.md 中是否有对应实施步骤 | 新增的漏斗维度、页面、API 端点 |
| 一致性 | 同一概念（漏斗维度、页面名称、技术栈、阈值、用户流程）跨文档细节是否完全吻合 | 数值、命名、枚举顺序的微妙差异 |
| 可实施性 | 按 session-prompts.md 执行后，产出是否完整实现另外两份文档描述的产品 | 隐式依赖、遗漏的配置步骤、顺序错误 |

3. 按以下格式输出每条发现（按严重度降序排列）：

### R{轮次}-G{序号} [{严重度}] {一句话标题}

- **类别**：缺失 | 矛盾 | 定义不充分
- **涉及文档**：{2-3 份文档名}
- **定位**：
  - `product-design.md` L{行号} / §{章节}
  - `ux-design.md` L{行号} / §{章节}
  - `session-prompts.md` L{行号} / §{章节}
- **预期**（引用权威文档原文）：{...}
- **实际**：{其他文档写了什么 / 未提及}

4. 如果发现零 Gap → 声明 ✅ 三份文档完全对齐，跳过阶段二，进入最终总结

阶段二：修复

对每个 Gap 执行第一性原理修复：

1. 溯源：根据权威性裁决规则，确定谁是真理来源
2. 最小编辑：用 Edit 工具将非权威文档向权威来源对齐——只改必须改的，不重写无关段落
3. 记录：每个修复写一行 ✅ R{轮次}-G{序号}: {文件名} L{行号} — {改了什么}

修复完所有 Gap 后，立即执行 /compact 压缩上下文，然后开始下一轮。

---
循环控制

```
repeat:
  Read docs/audit-prompt.md  ← 恢复完整指令
  阶段一（审计）
  if 零 Gap → break
  阶段二（修复）
  /compact
until 零 Gap OR 已完成 10 轮
```

- 每轮开头声明：`=== 第 N/10 轮审计 ===`
- 每轮结尾声明：`剩余 Gap: {数量}，{继续下一轮 / 循环结束}`
